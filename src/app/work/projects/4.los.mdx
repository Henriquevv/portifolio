## 1. O Problema de Negócio
A imprevisibilidade do Tempo de Permanência (LOS) é um dos maiores gargalos da operação hospitalar. Quando não sabemos quando o paciente terá alta, não conseguimos planejar o giro de leitos, o que gera superlotação no Pronto Socorro e cancelamento de cirurgias eletivas.

Este projeto foi desenvolvido como um estudo de caso em Ciência de Dados, simulando cenários reais para demonstrar como o uso de dados pode reduzir essas incertezas.

## 2. Proposta de Solução
O objetivo é sair da média simples (que esconde os casos graves) e utilizar Inteligência Artificial para apoiar a tomada de decisão clínica. Para isso, estruturei a análise em três frentes:

1.  Visibilidade do Fluxo (Análise de Sobrevivência): Mapear a probabilidade de alta do paciente dia após dia, permitindo prever a liberação de leitos.
2.  Triagem de Risco (Classificação): Identificar, já na admissão, quais pacientes têm alto risco de longa permanência (> 8 dias) para que o NIR/Gestão de Alta atue preventivamente.
3.  Diagnóstico de Causas (Inferência): Entender quais fatores (clínicos ou operacionais) estão "segurando" o paciente no hospital.

### 4. Construção da Base Analítica e Engenharia de Variáveis

Dados extraídos diretamente do sistema de gestão (ERP) costumam conter ruídos que distorcem indicadores. Nesta etapa, processei a base bruta para garantir que o modelo aprenda com a realidade operacional, e não com erros de registro.

Transformações Estratégicas:

* Saneamento Temporal: Padronização rigorosa das datas de entrada e saída. Isso evita que erros de digitação (ex: trocar dia por mês) criem "outliers artificiais" que prejudicariam a média de permanência.
* Cálculo de Ocupação Real (LOS): Definição do tempo de estadia ($\Delta t$). Ajustei casos de "alta no mesmo dia" para garantir que todo paciente contabilize consumo de recursos, eliminando valores nulos.
* Gestão de Pacientes Ativos (Censura): Um diferencial da Análise de Sobrevivência. Criei a variável de evento ($E$) para ensinar o modelo a distinguir:
    * $E=1$: Ciclo fechado (Paciente teve Alta/Óbito).
    * $E=0$: Censura (Paciente *ainda está no leito*). Isso impede que o modelo ache que uma internação longa em andamento é um erro ou um dado faltante.
* Estabilização Estatística (Log-Transform): Como o tempo de internação não segue uma curva normal (muitos pacientes rápidos, poucos muito demorados), apliquei uma transformação logarítmica (`np.log1p`). Isso equilibra o peso dos casos extremos, permitindo que o modelo aprenda os padrões da maioria sem ignorar os casos complexos.

```python
    df['Evento_ocorreu'] = np.where(df['DT_SAIDA'].notna(), 1, 0)

    data_referencia = df['DT_SAIDA'].max()
    if pd.isna(data_referencia):
        data_referencia = pd.Timestamp.now()

    mask_internado = df['DT_SAIDA'].isna()
    if mask_internado.any():
         df.loc[mask_internado, 'LOS'] = (data_referencia - df.loc[mask_internado, 'DT_ENTRADA']).dt.days

    df['LOS'] = df['LOS'].fillna(0).clip(lower=1)

    df['LOS_LOG'] = np.log1p(df['LOS'])

    if 'CD_CID_PRINCIPAL' in df.columns:
        df['CID_CAPITULO'] = df['CD_CID_PRINCIPAL'].astype(str).str[0].fillna('ND')
    else:
        df['CID_CAPITULO'] = 'ND'

    cols_demograficas = ['IDADE', 'SEXO']

        'CID_CAPITULO', 'CD_CID_PRINCIPAL',

    ]

    cols_flags = [c for c in cols_clinicas if c.startswith('FL_')]
    df[cols_flags] = df[cols_flags].fillna(0)

    cols_contexto = ['CARATER_ATENDIMENTO', 'ORIGEM_PACIENTE', 'CD_SETOR_ENTRADA', 'TIPO_SAIDA']
    cols_sinais_vitais = ['SV_PAS_ADMISSAO', 'SV_PAD_ADMISSAO', 'SV_FREQ_CARDIACA']

    todas_cols_desejadas = cols_demograficas + cols_clinicas + cols_contexto + cols_sinais_vitais
    cols_existentes = [c for c in todas_cols_desejadas if c in df.columns]

    colunas_finais = cols_existentes + ['DT_ENTRADA', 'DT_SAIDA', 'LOS', 'LOS_LOG', 'Evento_ocorreu']
    df_mestre = df[colunas_finais].copy()

    print("\n Dataset Mestre Criado")
    print("Dimensões:", df_mestre.shape)
    display(df_mestre[['CD_CID_PRINCIPAL', 'CID_CAPITULO', 'LOS', 'LOS_LOG']].head())
```

### 5. Comportamento da Permanência (EDA)

Antes de prever o futuro, é crucial entender o presente. Nesta etapa, analisei a distribuição real do Tempo de Permanência (LOS) para estabelecer nossa linha de base (baseline) de comparação.

Insights Estruturais:

* A "Armadilha da Média" (Distribuição Assimétrica): Confirmei que os dados hospitalares apresentam uma "cauda longa" (*right-skewed*). Poucos pacientes de longuíssima permanência puxam a média para cima, fazendo com que ela não represente o paciente típico.
    * *Ação Técnica:* Apliquei transformações logarítmicas para normalizar os dados. Isso impede que o modelo fique enviesado pelos *outliers* e erre a previsão da grande maioria dos atendimentos.

* A Régua de Comparação (Kaplan-Meier Global): Construí a Curva de Sobrevivência Global para responder à pergunta: *"Qual é a probabilidade estatística de um paciente qualquer continuar internado após o dia X?"*.
    * *Objetivo:* Esta curva atua como nosso "zero". Qualquer grupo de pacientes (ex: Idosos, Cirúrgicos) que se desviar muito desta curva será identificado como um ponto de atenção ou de eficiência.

### Diagnóstico do Comportamento da Permanência (EDA)

A análise exploratória serviu para validar a qualidade dos dados e, principalmente, para definir a estratégia de modelagem. Os dados revelaram que tratar a internação como uma distribuição normal simples levaria a erros de previsão.

Principais Descobertas:

#### A "Armadilha da Média"
Identifiquei uma discrepância crítica entre as métricas centrais:
* Média: 5.35 dias.
* Mediana: 2.00 dias.

Insight de Negócio: O paciente "típico" fica apenas 2 dias. A média é inflada (mais que o dobro da mediana) por poucos casos de longuíssima permanência. Utilizar a média simples para planejamento de capacidade geraria distorções graves, alocando recursos errados para a maioria dos pacientes.

#### Identificação da Cauda Longa (Assimetria)
O histograma confirmou que a operação lida com dois perfis de ocupação muito distintos:
* Giro Rápido (0-5 dias): A grande maioria dos casos.
* Cauda Longa (Extremos): Casos de alta complexidade que, embora menos frequentes, consomem desproporcionalmente os dias-leito disponíveis.

#### Correção Técnica (Transformação Logarítmica)
Para lidar com essa assimetria, apliquei a função `np.log1p` na variável alvo.
* Objetivo: Comprimir a escala dos valores extremos (dias muito altos) para aproximar a distribuição de uma curva normal. Isso impede que o algoritmo de regressão fique "confuso" tentando ajustar o erro apenas pelos casos extremos, melhorando a precisão geral.

Decisão Estratégica:
Dada a alta variabilidade confirmada, concluí que um modelo de regressão pura (prever o dia exato) terá limitações naturais. Portanto, a estratégia mais robusta para o hospital é atuar em duas frentes: estimar a duração, mas principalmente Classificar o Risco, segregando pacientes de "Giro Rápido" vs. "Longa Permanência" (> 8 dias).

### 6. Refinamento dos Dados e Criação de Inteligência

Para que o modelo enxergue padrões complexos, não basta entregar os dados brutos do sistema. Realizei um processo de Engenharia de Atributos para traduzir registros administrativos em fatores de risco tangíveis para o algoritmo.

Estratégias de Enriquecimento:

1.  Contexto Sazonal (`IS_INVERNO`):
    Criei um indicador específico para o período de junho a setembro. Isso ensina o modelo que, nestes meses, a complexidade dos casos aumenta naturalmente (devido a doenças respiratórias), ajustando a previsão de permanência para cima independente do diagnóstico base.

2.  Saneamento de Sinais Vitais:
    Implementei filtros de segurança para dados fisiológicos. Valores impossíveis (ex: Pressão Arterial ou Saturação zeradas) foram tratados como erro de aferição/digitação e corrigidos estatisticamente. Isso impede que o modelo confunda um erro de cadastro na triagem com um paciente em óbito ou alta gravidade.

3.  Tradução de Dados Clínicos:
    Algoritmos matemáticos não interpretam texto. Converti variáveis qualitativas (como Especialidade e Convênio) em representações numéricas binárias (*One-Hot Encoding*), permitindo que o modelo pondere o peso exato de cada especialidade no tempo de ocupação.

4.  Protocolo de Validação (Split Estratificado):
    Separei 20% dos dados para teste. A divisão foi feita garantindo que a proporção de pacientes de alta vs. internados fosse idêntica nos dois grupos, simulando com fidelidade o mix de pacientes que o hospital recebe no dia a dia.

### 7. Estimativa de Dias-Leito: Modelagem de Regressão

Nesta etapa, o objetivo foi responder à pergunta operacional mais crítica: *"Quantos dias, exatamente, este paciente deve ocupar o leito?"*. Tratei o problema como uma tarefa de regressão para estimar o LOS contínuo.

Decisões Estratégicas de Modelagem:

* O Motor (Random Forest): Optei por um modelo de *ensemble* (floresta aleatória) pela sua capacidade de entender a complexidade humana.
    * *Por que:* A saúde não é linear. A combinação de fatores (ex: *Idade Avançada* + *Insuficiência Cardíaca*) gera um risco exponencial, não apenas uma soma simples. Modelos lineares falhariam em capturar essa nuance, mas árvores de decisão são excelentes nisso.

* A Métrica de Sucesso (MAE - Erro Médio): Para avaliar o modelo, priorizei o Erro Absoluto Médio.
    * *Justificativa:* Como vimos na análise exploratória, temos casos extremos (internações de meses). Métricas tradicionais (como RMSE) dariam peso excessivo a esses casos raros, distorcendo a precisão para o paciente "típico" de 3 dias. O MAE mantém o foco na performance do dia a dia.

Gestão de Expectativa:
A recuperação biológica possui variáveis imprevisíveis (estocásticas). O objetivo deste modelo não é "adivinhar o futuro" com precisão de minutos, mas minimizar o erro médio para fornecer um norte de planejamento confiável para a equipe de gestão de leitos.


```python
import numpy as np
import pandas as pd
import matplotlib.pyplot as plt
import seaborn as sns
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import OneHotEncoder, RobustScaler
from sklearn.impute import SimpleImputer
from sklearn.ensemble import RandomForestRegressor
from sklearn.metrics import mean_absolute_error, mean_squared_error, r2_score


%store -r features_num features_cat features_bin


numeric_transformer = Pipeline(steps=[
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler', RobustScaler())
])

categorical_transformer = Pipeline(steps=[
    ('imputer', SimpleImputer(strategy='constant', fill_value='Desconhecido')),
    ('onehot', OneHotEncoder(handle_unknown='ignore', sparse_output=False))
])

preprocessor = ColumnTransformer(
    transformers=[
        ('num', numeric_transformer, features_num),
        ('cat', categorical_transformer, features_cat),
        ('bin', 'passthrough', features_bin)
    ])

model_rf = RandomForestRegressor(
    n_estimators=200,
    max_depth=15,
    min_samples_leaf=4,
    n_jobs=-1,
    random_state=42
)

melhor_modelo = Pipeline(steps=[('preprocessor', preprocessor),
                                ('model', model_rf)])

print("Treinando Random Forest...")
melhor_modelo.fit(X_train, y_days_train)

y_pred = melhor_modelo.predict(X_test)

mae = mean_absolute_error(y_days_test, y_pred)
rmse = np.sqrt(mean_squared_error(y_days_test, y_pred))
r2 = r2_score(y_days_test, y_pred)

print(f"\nRESULTADOS FINAIS (Random Forest):")
print(f"MAE (Erro Médio): {mae:.2f} dias")
print(f"RMSE: {rmse:.2f}")
print(f"R²: {r2:.4f}")


plt.figure(figsize=(10, 6))
sns.scatterplot(x=y_days_test, y=y_pred, alpha=0.4, color='purple')
plt.plot([0, 150], [0, 150], '--r', linewidth=2, label='Ideal')
plt.title('Previsão vs Real (Random Forest)')
plt.xlabel('Dias Reais')
plt.ylabel('Dias Previstos')
plt.xlim(0, 60)
plt.ylim(0, 60)
plt.legend()
plt.show()
```

### Discussão dos Resultados: Limites da Previsão Exata

Nesta etapa, avaliei a capacidade do modelo de regressão (*Random Forest*) em responder à pergunta: *"Em que dia exato este paciente terá alta?"*. Os resultados mostram uma clara dicotomia entre pacientes de rotina e casos complexos.

Diagnóstico de Performance:

#### O "Ponto Cego" da Admissão (R² ≈ 18%)
O modelo consegue explicar 18.4% da variabilidade do tempo de internação.
* Interpretação de Negócio: Isso não é um defeito do modelo, mas uma característica do setor. Dados de admissão (quem é o paciente) definem o início da jornada, mas o que realmente determina a alta são os eventos intra-hospitalares (resposta a medicamentos, disponibilidade de exames, intercorrências) que ainda não aconteceram.

#### Eficiência no Paciente Típico (MAE vs. RMSE)
As métricas de erro contam duas histórias diferentes:
* Para o dia a dia (MAE = 3.67 dias): O erro médio absoluto é baixo. Para a vasta maioria dos pacientes (casos de baixa/média complexidade), o modelo é uma ferramenta confiável para planejamento de fluxo.
* Para casos extremos (RMSE = 9.45 dias): O erro quadrático salta devido aos *outliers*. O modelo perde precisão ao tentar quantificar internações muito longas (ex: prever se ficará 40 ou 60 dias).

#### Análise Visual: Conservadorismo em Casos Graves
O gráfico de dispersão revela que o algoritmo é conservador:
* Em casos de longa permanência, ele tende a subestimar o tempo real (prevê menos dias do que realmente ocorre).
* Diagnóstico: O modelo consegue detectar *que* o paciente vai demorar, mas tem dificuldade em precisar *o quanto*.

Pivô Estratégico:
Insistir em prever o dia exato de alta para pacientes graves traz um risco de frustração para a equipe assistencial. A decisão mais inteligente é mudar a pergunta. Em vez de *"Que dia ele sai?"*, focaremos em *"Este paciente é de Alto Risco (> 8 dias)?"*.

Isso nos leva à próxima etapa: Classificação de Risco, onde transformamos a incerteza da regressão em um sistema de alerta binário para o NIR (Núcleo Interno de Regulação).

### 8. Identificando Gargalos: Comparação de Perfis Clínicos

Sabemos que "não existe o paciente médio". Um paciente eletivo de cirurgia simples não se comporta como um paciente agudo de pronto-socorro. Nesta etapa, utilizei o estimador Kaplan-Meier para segmentar a curva de sobrevivência e provar onde estão os gargalos de ocupação.

Estratégia de Análise:

* Visualização de Fluxos (Curvas Estratificadas):
    Plotei curvas de alta separadas para diferentes grupos (ex: *Eletivo vs. Urgência*).
    * *O que buscamos:* Se as curvas se afastarem visualmente, significa que um grupo gira o leito muito mais rápido que o outro, exigindo metas de gestão distintas.

* Prova Matemática (Teste de Log-Rank):
    Não basta o gráfico parecer diferente; é preciso ter certeza estatística.
    * *Aplicação:* Utilizei o teste de Log-Rank para calcular se a diferença no tempo de internação entre os grupos é real ou fruto do acaso.
    * *Decisão:* Se o p-valor for < 0.05, confirmamos cientificamente que aquele fator (ex: Diagnóstico) é um determinante crítico para a demora da alta.


```python
from lifelines import KaplanMeierFitter
from lifelines.statistics import multivariate_logrank_test
```

### Discussão dos Resultados: Onde estão os Gargalos?

A segmentação via Kaplan-Meier revelou que o hospital opera, na prática, com "dois hospitais" dentro de um só, cada um com dinâmicas de ocupação radicalmente opostas.

Diagnóstico Operacional:

#### A Dicotomia do Atendimento (Urgência vs. Eletivo)
As curvas de sobrevivência expõem duas realidades distintas:
* Fluxo Eletivo (Alta Eficiência): Curva com queda abrupta nas primeiras 48h.
    * *Insight:* Procedimentos programados giram o leito rápido e com previsibilidade. São a "locomotiva" da receita e eficiência.
* Fluxo de Urgência (Gargalo): Curva suave com cauda longa.
    * *Insight:* Pacientes não programados têm mediana de internação superior (~4 dias) e alta variabilidade. São os principais ofensores da ocupação, exigindo gestão de caso individualizada.

#### O Peso das Doenças Crônicas (Análise por CID)
Identifiquei que o diagnóstico de entrada dita o ritmo da alta:
* Retentores de Leito: Capítulos 'I' (Circulatório) e 'J' (Respiratório) apresentaram as piores curvas de alta. Pacientes com descompensações cardíacas ou pulmonares são estruturalmente mais lentos para recuperar, exigindo protocolos de desospitalização específicos.
* Giro Rápido: Em contraste, o capítulo 'K' (Digestivo) mostrou altas rápidas ($t < 2$ dias), validando o perfil cirúrgico/resolutivo.

#### Validação Científica
O teste de *Log-Rank* confirmou ($p < 0.0001$) que essas diferenças não são aleatórias.
* Conclusão de Negócio: A origem do paciente (Urgência) e a doença base são determinantes críticos. Um modelo preditivo que ignore essas variáveis será cego para a realidade do hospital. Validamos, portanto, que elas devem ser as principais *features* (variáveis preditoras) na etapa de Machine Learning.

### 9. Solução de Triagem: Identificação de Pacientes Críticos

Diante da dificuldade em prever o dia exato de alta para casos complexos (o "ponto cego" da regressão), adotei uma abordagem complementar: a Discretização do Risco.

Se não podemos garantir *quando* o paciente sai, precisamos garantir que o hospital saiba *quem* são os pacientes que vão ficar muito tempo.

Definição da Nova Estratégia:

* A Pergunta de Negócio (Target Binário): O modelo deixa de tentar acertar um número e passa a calcular a probabilidade de um cenário crítico: *"Este paciente tem chance de ultrapassar 8 dias de internação?"*.
    * *Por que 8 dias?* Este valor representa o quartil superior da nossa distribuição. Pacientes que superam essa marca são os maiores ofensores do giro de leitos.

* O Motor (XGBoost Classifier): Utilizei o algoritmo XGBoost, padrão da indústria para classificação em dados desbalanceados (onde os casos de "sucesso" são a minoria), para separar o joio do trigo.

Critério de Sucesso: A Política de "Erro Seguro"
Em gestão hospitalar, nem todo erro tem o mesmo peso.
* O Erro Aceitável (Falso Positivo): O modelo diz que o paciente é grave, a equipe audita, mas ele tem alta rápida. *Custo: Tempo administrativo.*
* O Erro Inaceitável (Falso Negativo): O modelo diz que o paciente é simples, ninguém olha, e ele fica 20 dias ocupando o leito sem planejamento. *Custo: Bloqueio de leito e desassistência.*

Decisão: Otimizei o modelo focando em Recall (Sensibilidade). O objetivo é criar uma "rede de segurança" que capture o máximo possível de pacientes graves, mesmo que isso gere alguns alertas falsos para checagem.

### Validação e Calibragem Operacional

Um modelo preditivo em saúde não pode apenas ser matematicamente preciso; ele precisa ser seguro. Nesta etapa, submeti o classificador a testes de estresse e ajustei sua sensibilidade para garantir que nenhum paciente grave passe despercebido.

Resultados da Validação:

#### Confiabilidade do Modelo (Estabilidade)
Antes de colocar em prática, verifiquei se o modelo era consistente.
* Resultado: O modelo manteve alta performance (AUC ~0.86) em 5 cenários de teste diferentes, com variação mínima.
* O que isso significa: O algoritmo é robusto. Ele não "teve sorte" com um grupo específico de pacientes; ele aprendeu padrões reais que se repetem, oferecendo segurança para uso diário.

#### A Regra do "Erro Seguro" (Ajuste de Corte)
Por padrão, algoritmos esperam 50% de certeza para emitir um alerta. Para o ambiente hospitalar, considerei isso arriscado.
* Decisão de Negócio: Reduzi o gatilho de alerta para 35% de probabilidade.
* Impacto: Com esse ajuste, aumentamos a sensibilidade do modelo (Recall) de 76% para 84%. Na prática, isso cria uma "malha fina" que captura a vasta maioria dos casos complexos, evitando que pacientes graves fiquem sem gestão.

#### O Custo da Segurança (Trade-off)
Ao aumentar a sensibilidade, aceitamos naturalmente mais alertas.
* A Realidade: De cada 3 alertas de "Alto Risco", 1 será de fato uma longa permanência crítica.
* Justificativa: O custo operacional de a equipe checar 2 pacientes a mais (falsos positivos) é irrisório comparado ao prejuízo financeiro e assistencial de deixar um paciente crítico ocupar um leito por semanas sem planejamento.

Conclusão:
O modelo está calibrado para atuar como um Radar de Alta Sensibilidade. Ele prioriza a segurança, garantindo que a equipe de gestão de leitos receba alertas precoces sobre os pacientes que mais demandarão recursos.

* Diretriz: Podemos criar um "Fast Track" administrativo para esses pacientes. A equipe pode reduzir drasticamente o tempo gasto monitorando esses casos, liberando horas de trabalho para focar nos casos complexos.

#### O Custo da Atenção (VPP = 31.7%)
Para garantir que não perderíamos pacientes graves, aceitamos que o modelo dispare mais alertas.
* A Realidade: De cada 3 alertas vermelhos, 1 será crítico e 2 serão alarmes falsos (pacientes que, apesar do perfil de risco, terão alta rápida).
* Diretriz: O alerta dispara uma Sondagem, não uma intervenção. O custo de checar 2 pacientes a mais é o "preço do seguro" que pagamos para garantir a detecção daquele 1 paciente crítico.

Veredito:
A ferramenta funciona segregando o fluxo em dois: uma via expressa de baixo monitoramento (para a maioria) e uma lista de atenção focal (para os alertas), otimizando a alocação da equipe de gestão de leitos.
